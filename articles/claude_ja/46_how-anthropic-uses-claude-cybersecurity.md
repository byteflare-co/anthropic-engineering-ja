# Anthropic のサイバーセキュリティチームが Claude Code で脅威検知プラットフォームを構築した方法

Jackie Bow はキャリアを通してずっと、本当に重要なコンテキスト——ログやアラートだけではなく、Slack の会話、社内ドキュメント、組織知といった、何かが本物の脅威なのか単なるノイズなのかを判断するための情報——にアクセスできるツールを思い描いてきました。しかも、人間がデータを手作業でかき分ける負担を背負う必要のないツールです。

Anthropic に入社して、彼女はついにそれらを構築するチャンスを手にしました。Claude を共同作業者として、です。Jackie は Anthropic の Detection Platform Engineering チームを率いており、防御的サイバーセキュリティ、すなわち脆弱性を探し回るのではなく、脅威の検知と侵害の可能性への対応に注力しています。仕事の中身は、不審な挙動を求めてシステムを監視し、セキュリティアラートをトリアージし、インシデントになる前に異常を調査することです。

ますます高い能力を持つ AI モデルを構築している企業にとって、この仕事は土台になるものです。[Anthropic の Responsible Scaling Policy](https://www.anthropic.com/news/anthropics-responsible-scaling-policy) はプロダクトのリリースをセキュリティ上のコミットメントに直接結びつけており、Bow のチームは会社が何を安全に世に出せるかを左右する役割を担っています。

「今はセキュリティエンジニアにとっての黄金時代だと感じています」と、Anthropic の Detection Platform Engineering チームの Technical Lead を務める Bow は言います。「ずっと欲しいと思ってきたツールを、ついに自分で作れるようになりました。」

## **問題: データとアラートに溺れる**

セキュリティリーダーであれば、このパターンを嫌というほど知っているはずです。

アラートが鳴る。アナリストはターミナルを開き、いつもの儀式を始めます。5 つも 6 つもの異なるツールの間を飛び回り、それぞれが独自のクエリ言語とメンタルモデルを要求してきます。複数のプラットフォームにまたがって専門知識を維持しつつ、異なるインタフェースとクエリ構文の間で絶えずコンテキストスイッチを強いられます。

それぞれの調査は、つながっていない複数のシステムに散らばった断片をつなぎ合わせる「データ考古学」の作業になります。多くのチームにとって、シンプルな調査でも数時間を消費し、複雑なものでは数日に及ぶこともあります。

「人間が 1 日に見られるアラートには限度があって、それを超えると詳細さがどうしても落ちていくんです」と Jackie は説明します。

彼女のチームはこれをなんとかしようと決めました。自分たちに問いかけたのは、こうした問いです。私たちのエネルギーを奪っているのは何か？ 何が繰り返し作業に感じられるか？ インパクトのある仕事を阻んでいるのは何か？

答えは明確でした。アラートのトリアージは、それが本物の脅威かどうかを判断する前に何時間も消費してしまうことがあります。つながっていないシステム間での手作業の相関付けがすべてを遅くしていました。そして、クエリ言語やインタフェース間の絶え間ないコンテキストスイッチが認知的なオーバーヘッドを生み、それが 1 日を通して積み重なっていきました。

会社が成長するにつれて、攻撃対象領域も拡大し、セキュリティチームへの要求も増えていきます。

「Claude のようなものを使って自分たちを拡張しなければ、Anthropic のニーズに合わせてスケールすることはできません」と Jackie は語ります。

## **解決策: Claude Looks Up Evidence (CLUE)**

数か月にわたって、Bow のチームは CLUE を構築しました。これは、セキュリティチームによる脅威調査のやり方を再構想する検知・対応プラットフォームです。スタックにダッシュボードをもう一つ追加するのではなく、CLUE は Claude を動力源とする自然言語インタフェースを提供し、tool use を通じて Anthropic の社内システムに直接接続します。

CLUE を Claude Code とともに構築したことで、従来のソフトウェア開発のタイムラインは指数関数的に圧縮され、日々の業務に取り組みながら、このシステムを構築する余裕が生まれました。Bow のチームは 1 日で proof of concept を動かしました。設計ドキュメント、開発ステップ、実装は 1 週間以内に完了しました。

「私たちが作ったものの多くは、Claude Code と対話して生まれたものです」と Jackie は振り返ります。「設計パートナーであると同時に、共同作業者でもありました。」

彼女の認識を変える瞬間は、Claude Code に CLUE のインタフェースにボタンを追加してほしいと頼んだときに訪れました。JavaScript フレームワークと CSS のデバッグというお馴染みの苦行が始まると覚悟していました。Claude Code はその場でその機能を実装し、しかも彼女自身がやるよりも上手にやってのけました。

「そのときに気付いたんです。私はもう、自分の技術的な限界に縛られていない、と。思いつく限りのものを作れるんだ、と」と彼女は言います。

### **CLUE Triage**

アラートが流れ込んでくると、CLUE Triage は人間のアナリストが目にする前に一次トリアージを実行します。Claude はツールを使って、Slack メッセージ、社内ドキュメント、コードリポジトリ、データウェアハウスなど、Anthropic のシステム全体から追加のコンテキストを集めて、各アラートをエンリッチします。そして、false positive、true positive、悪意あり、想定された挙動、のいずれかのディスポジションを割り当てます。アラートごとに信頼度スコアが付くので、アナリストはどこに注意を集中すべきかが分かります。

このエンリッチメントのステップは、どのセキュリティアナリストも知っている問題を解決します。アラートは孤立したシグナルとしてやってきます。ログイン失敗、変わった API 呼び出し、設定の変更——コンテキストがなければ、これらは単なるノイズです。コンテキスト（そのユーザーが誰なのか、何の作業をしていたか、これがチームのパターンと合致するかどうか）があれば、明確なシグナルが浮かび上がります。

「その社内コンテキストこそ、アラートを自分たちの環境に即して文脈付けるための、失われていたピースなのです」と Jackie は説明します。

### **CLUE Investigate**

CLUE があれば、セキュリティアナリストはセキュリティクリティカルなすべてのログに対して自然言語でクエリできるようになります。「過去 1 日のうち、このシステムへのログイン失敗をすべて教えて」と知りたい？ CLUE に頼むだけで、Claude が必要な SQL クエリを実行してくれます。

「Claude は人間より、はるかに精密なクエリを書くのが得意です」と Jackie は語ります。このツールはエージェント的なループで動きます。オーケストレーターがサブエージェントにコマンドを発行し、サブエージェントは並列にクエリを実行し、知見を集め、結果をまとまった調査サマリーへと統合します。手作業の相関付けで何時間もかかっていた作業が、いまでは 3〜4 分で終わります。

数字もそれを裏付けています。CLUE はセッションあたり平均で 25 回のツール呼び出しと、ほぼ 11 回のクエリを実行しています。これはアナリストが手作業で実行できる量をはるかに上回り、しかも精度も高いものです。これらのツール呼び出しの一つひとつは、本来であれば別のコンソールを開いたり、別のインタフェースに切り替えたりすることを要求していたはずです。

ただし、本当の差別化要因は社内コンテキストです。CLUE はツールを通じて Anthropic のシステムに直接接続しており、外部のセキュリティプラットフォームが決してアクセスできないであろう組織知に、Claude がアクセスできるようになっています。アラートが鳴ると、Claude は Slack をチェックして、チームが計画していたメンテナンスについて議論していなかったかを確認できます。データウェアハウスをクエリして、ベースラインの挙動を理解することもできます。コードリポジトリを調べて、サービスが実際に何をするものなのかを把握することもできます。

### **データガバナンスレビュー**

チームは、よくあるデータガバナンスのシナリオを使って CLUE の能力をデモしました。3 人の業務委託契約者が、過去 2 か月間にアクセスすべきでなかった文書にアクセスしていないかを確認する、というものです。

Bow によれば、この種の調査は通常、少なくとも半日の手作業が必要になります。アクセスログのクエリ、権限のクロスリファレンス、文書分類のレビューといった作業です。CLUE を使えば、Claude がリクエストを読み、計画を立て、技術的な複雑性を抽象化した冗長なクエリを生成します。調査は数分で完了し、サマリーとレコメンデーションが出力されるとともに、実行されたすべてのクエリへの完全な透明性が保たれます。

## **インパクトを測る**

チームが CLUE の構築に着手したとき、彼らは単に速く感じるかどうかだけでなく、結果を定量化できるかどうかを理解したいと考えました。

**false positive の削減**: CLUE Triage の導入前は、おおよそ 3 件に 1 件のアラートが false positive でした。その比率は 7% にまで下がり、アナリストは意味のあるシグナルに時間を使えるようになりました。\*

**より広範なカバレッジ**: 速度よりも重要かもしれないのは、チームが今や検証できる対象です。CLUE の前は、信頼度の低いシグナルは時間がなくて検証されないままになっていました。いまでは、CLUE Triage が入ってくるすべてのアラートをエンリッチメント付きで処理し、バッチ処理は以前ならダッシュボード上でノイズになっていたはずの何千ものシグナルを扱います。

**スケールでの時間節約**: 30 日間の利用に基づくと、CLUE はおよそ 12,000 件のクエリと 27,000 件のツール呼び出しを自動化しました。手作業で行うとおよそ 1,870 時間（234 人日）かかると見積もられる作業量です。これは、手作業のトリアージと比較して 5〜10 倍の時間節約に相当します。\*

**まだ測り方を学んでいるもの**: 精度は速度よりも定量化が難しい指標です。チームは CLUE のディスポジションをレビューし、意見が分かれたものを追跡していますが、Claude がアナリストの見落としをどれだけ拾えているか——あるいは逆もまた然りか——を理解するためのフィードバックループは、まだ構築中です。トランスクリプトが助けになります。すべての調査について、Claude が何を調べ、どのようにして結論にたどり着いたかを正確に監査できるからです。

## **これからの方向性: Claude を Claude らしく調査させる**

AI 研究には「[the bitter lesson](http://www.incompleteideas.net/IncIdeas/BitterLesson.html)」と呼ばれる考え方があります。人間特有の推論をモデルにエンコードする方法は、モデルに汎用的な能力を与えて自分なりのアプローチを見つけさせる方法に、一貫して劣る、という観察です。Bow と彼女のチームは、これが検知と対応にとって何を意味するかを考えてきました。

「CLUE の開発初期には、Claude の調査経路をどこまで制約するかをチームで議論していました」と Bow は言います。「SOAR 時代の本能は、こう言ってきます。プレイブックを作れ、すべてのステップを定義しろ、プロセスを決定論的にしろ、と。でも、私たちはあることに気付き続けていました。Claude に探索の余地を与え——固定されたシーケンスではなく、ツールへのアクセスとゴールだけを与え——ると、私たちが指示しなかったような調査経路をたどることがよくあったのです。そして時には、そうした経路が、私たちなら見落としていただろうコンテキストを浮かび上がらせました。」

鍵は、Claude に境界（どのツールを使えるか、どのデータにアクセスできるか）を与えつつ、戦略はオープンなままにしておくことです。この洞察が、CLUE を次にどこへ持っていくかを形作っています。

**リアクティブからプロアクティブへ**: 現在、CLUE はアラートに反応する形で動いています。イベントが発火して、Claude が調査します。しかし、アーキテクチャはより野心的なものを支えられるようになっています。継続的な探索です。検知ルールがトリガされるのを待つのではなく、Claude エージェントが能動的に不審なパターンを狩りに行く——チームが書いたどのルールにもマッチしない異常や、個別に見ると正常に見えるのに集約すると異常に見える挙動を、です。

**自分自身から学ぶ**: チームはすべての調査トランスクリプトを保存しています。そのコーパスは、過去の調査がどのように展開したかのパターンを Claude が問い合わせられるナレッジベースになりつつあります。時間が経つにつれて、CLUE は人間のアナリストには到底維持できないような組織的な記憶を育てていきます。

**非決定性を受け入れる**: 従来のセキュリティツーリングは不整合をバグとして扱います。CLUE はそれを機能として扱います。同じアラートが、日によって違うように調査されることもあり得るし、それで構わないのです。二番目の経路が、最初の経路では見落とした何かを見つけることもあるからです。チームは、複数の調査戦略を並列に走らせて結果を比較する実験を進めています。

「セキュリティオペレーションにとっての the bitter lesson とは何か？ 私たちは長年、人間がどう調査するかをエンコードしたシステムを作ってきました。次世代のツールは、モデルに調査する能力を与え、私たちが指示するよりも良いアプローチを見つけさせるべきなのです」と Bow は付け加えます。
