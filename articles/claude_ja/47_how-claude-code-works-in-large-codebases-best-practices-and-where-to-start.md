# Claude Code が大規模コードベースで動く仕組み: ベストプラクティスとどこから始めるか

Claude Code は、数百万行規模のモノレポ、数十年来のレガシーシステム、数十のリポジトリにまたがる分散アーキテクチャ、そして数千人規模の開発者を抱える組織のプロダクション環境で動いています。こうした環境には、より小さくシンプルなコードベースには存在しない難しさがあります。サブディレクトリごとに異なるビルドコマンドだったり、共有ルートを持たないフォルダ群に散らばったレガシーコードだったりです。

本記事では、私たちが観察してきた、大規模環境で Claude Code の導入を成功に導いたパターンを取り上げます。「大規模コードベース」という言葉で私たちが指すのは、幅広いデプロイメントのことです——数百万行規模のモノレポ、何十年にもわたって築かれたレガシーシステム、別々のリポジトリにまたがる数十のマイクロサービス、あるいはその組み合わせなどです。そこには、C、C++、C#、Java、PHP のように、チームが必ずしも AI コーディングツールと結びつけて考えてこなかった言語で動くコードベースも含まれます。（最近のモデルリリースを経て、Claude Code はそうした言語でもチームの想像以上にうまく動きます。）大規模コードベースの導入はそれぞれ、固有のバージョン管理、チーム構造、積み上がってきた慣習に形づくられますが、ここで紹介するパターンは横断的に通用しますし、Claude Code の導入を検討するチームにとって良い出発点になります。

## Claude Code が大規模コードベースをどう辿るか

Claude Code は、ソフトウェアエンジニアと同じやり方でコードベースを辿ります。ファイルシステムを歩き回り、ファイルを読み、grep を使って必要なものを正確に見つけ、コードベース全体にわたる参照を追いかけます。動作するのは開発者のマシン上のローカルであり、コードベースのインデックスを構築・保守したりサーバーにアップロードしたりする必要はありません。

これまでの AI コーディングツールは、コードベース全体を埋め込み、クエリ時に関連チャンクを取ってくる RAG ベースの検索に依存していました。大規模になると、こうしたシステムは破綻しがちです。埋め込みパイプラインが、活発に動いているエンジニアリングチームのペースに追いつけないからです。開発者がインデックスにクエリを投げる頃には、それは何日も、何週間も、あるいは何時間も前のコードベースを反映したものになっています。その結果、検索は 2 週間前にチームがリネームした関数を返してきたり、先週のスプリントで削除されたモジュールを参照したりするものの、それらが古い情報だという表示は一切ありません。

エージェント的な探索は、こうした失敗モードを回避します。何千人ものエンジニアが新しいコードをコミットしていく中で、保守すべき埋め込みパイプラインや中央集権的なインデックスは存在しません。各開発者のインスタンスは、ライブのコードベースを相手に動きます。

ただし、このアプローチにはトレードオフがあります——どこを見ればよいか分かるだけの開始時コンテキストが Claude にあるときに最もうまく機能するということです。つまり、Claude のナビゲーションの質は、コードベースのセットアップがどれだけ良くできているか、CLAUDE.md ファイルや skill によってコンテキストがどれだけ重ね合わせられているかに左右されます。10 億行規模のコードベースで漠然としたパターンの全インスタンスを探すよう Claude に頼めば、作業を始める前にコンテキストウィンドウの上限にぶつかるでしょう。コードベースのセットアップに投資したチームは、より良い結果を手にしています。

## ハーネスはモデルと同じくらい重要

Claude Code に関してよくある誤解のひとつは、その能力が使用するモデルだけで決まる、というものです。チームはモデルのベンチマークやテストタスクでの挙動に注目しがちです。しかし実際には、モデルの周囲に作られたエコシステム——ハーネス——のほうが、Claude Code の性能をモデル単体よりも左右します。

ハーネスは、それぞれ役割の異なる 5 つの拡張ポイント——CLAUDE.md ファイル、hook、skill、プラグイン、MCP サーバー——から成ります。チームがこれらを組み立てる順番は重要で、各レイヤーは前のレイヤーの上に積み上がっていきます。さらに 2 つの能力、LSP 連携とサブエージェントが、セットアップを完成させます。以下では、それぞれのコンポーネントと能力が何をするかを説明します。

[**CLAUDE.md**](https://code.claude.com/docs/en/memory) **ファイルが最初に来ます**。これらはセッション開始時に Claude が自動的に読み込むコンテキストファイルです——全体像のためのルートファイル、ローカルな慣習のためのサブディレクトリファイルです。Claude が何かをうまくこなすために必要な、コードベースの知識を与えてくれます。タスクによらず毎セッションで読み込まれるため、広く適用される事柄に絞り込んでおくことが、性能の足かせにしないために重要です。

[**Hook**](https://code.claude.com/docs/en/hooks-guide) **はセットアップを自己改善的にします**。多くのチームは、hook を Claude に何か誤った行動を取らせないためのスクリプトと考えがちですが、より価値のある使い方は継続的な改善です。stop hook は、セッション中に何が起きたかを振り返り、文脈が新鮮なうちに CLAUDE.md の更新を提案できます。start hook は、チーム固有のコンテキストを動的に読み込み、各開発者が手動設定なしに自分のモジュール向けの適切なセットアップを得られるようにします。lint やフォーマットのような自動チェックでは、hook はルールを決定論的に強制し、Claude に指示を覚えておかせるよりも一貫した結果を生み出します。

[**Skill**](https://code.claude.com/docs/en/skills) **は、適切な専門知識を必要なときだけ手元に置き、毎セッションを膨れ上がらせずに済ませてくれます**。タスク種別が数十にわたる大規模コードベースでは、すべての専門知識を毎セッションで存在させる必要はありません。skill はこれを [progressive disclosure](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices) を通じて解決します。本来であればコンテキスト枠を奪い合うことになる専門的なワークフローやドメイン知識を外に出しておき、タスクが求めたときだけ読み込ませるのです。たとえば、Claude が脆弱性のためにコードを評価するときにはセキュリティレビュー用 skill が読み込まれ、コード変更が行われてドキュメントの更新が必要になったときには文書処理用 skill が読み込まれる、といった具合です。

skill は特定のパスにスコープを絞り、コードベースの該当部分でのみアクティブになるようにもできます。決済サービスを所有するチームは、自分たちのデプロイ用 skill をそのディレクトリに紐づけておけば、モノレポ内の別の場所で誰かが作業しているときに自動読み込みされることはありません。

[**プラグイン**](https://code.claude.com/docs/en/plugins) **はうまくいくものを配布します**。大規模コードベースの難しさのひとつは、*良い*セットアップが部族的な知識のまま留まってしまうことです。プラグインは skill、hook、MCP 設定をひとつのインストール可能なパッケージにまとめます。新しいエンジニアが初日にそのプラグインをインストールすれば、すでに Claude を使ってきた人たちと同じコンテキストと能力を即座に手にできます。プラグインの更新は、[マネージドマーケットプレイス](https://support.claude.com/en/articles/13837433-manage-claude-cowork-plugins-for-your-organization)を通じて組織全体に配布できます。

たとえば、私たちが協働している大手小売企業は、Claude を社内アナリティクスプラットフォームに接続する skill を構築し、ビジネスアナリストが自分のワークフローを離れずにパフォーマンスデータを取り出せるようにしました。同社は、業務部門への広範なロールアウトに先立って、それをプラグインとして配布しました。

**Language server protocol (LSP) 連携は、開発者が IDE で得ているのと同じナビゲーションを Claude に与えます**。大規模コードベースの IDE では、多くの場合すでに LSP が動いており、「定義へ移動」や「すべての参照を検索」を支えています。これを Claude に露出してやると、シンボルレベルの精度が得られます——関数呼び出しを定義まで辿り、複数ファイルにまたがる参照をトレースし、異なる言語にまたがって同名の関数を区別できます。LSP がなければ、Claude はテキストでパターンマッチを行い、間違ったシンボルに辿り着くことがあります。私たちが協働したあるエンタープライズソフトウェア企業は、Claude Code の展開に先立って LSP 連携を全社的に導入しました。狙いは、C と C++ のナビゲーションを大規模に信頼できるものにすることでした。多言語コードベースにおいては、これは最も投資対効果の高い施策のひとつです。

**MCP サーバーはすべてを拡張します**。MCP サーバーは、それ以外の方法では到達できない社内ツール、データソース、API に Claude を接続する仕組みです。最も洗練されたチームは、構造化検索を Claude が直接呼び出せるツールとして公開する MCP サーバーを構築していました。それ以外には、Claude を社内ドキュメント、チケッティングシステム、アナリティクスプラットフォームに接続する例があります。

[**サブエージェント**](https://code.claude.com/docs/en/sub-agents) **は探索と編集を分離します**。サブエージェントは独自のコンテキストウィンドウを持つ独立した Claude インスタンスで、タスクを受け取って作業を行い、最終結果だけを親に返します。ハーネスが整っていれば、一部のチームは読み取り専用のサブエージェントを立ち上げてサブシステムをマッピングさせ、結果をファイルに書き出させたうえで、メインエージェントに全体像を踏まえた編集を行わせる、といった使い方をしています。

*Claude Code の拡張レイヤーの一覧。*

下の表は、各コンポーネントが何をするか、いつ読み込まれるか、そして私たちがよく目にする勘違いをまとめたものです。

| コンポーネント | 何か | いつ読み込まれるか | 向いていること | よくある混同 |
| --- | --- | --- | --- | --- |
| CLAUDE.md | Claude が自動的に読み込むコンテキストファイル | 毎セッション | プロジェクト固有の慣習、コードベースの知識 | skill に置くべき再利用可能な専門知識をここに入れてしまう |
| Hook | 重要な瞬間に実行されるスクリプト | イベントによってトリガー | 一貫した挙動の自動化、セッションの学びの取り込み | 自動実行されるべきことにプロンプトを使う |
| Skill | 特定のタスク種別向けのパッケージ化された指示 | 関連するときにオンデマンドで | セッションやプロジェクトをまたいで再利用できる専門知識 | 代わりに CLAUDE.md にすべてを詰め込む |
| プラグイン | バンドルされた skill、hook、MCP 設定 | 一度設定すれば常時利用可能 | 動くセットアップを組織全体に配布する | 良いセットアップを部族的な知識のままにしておく |
| Language server protocol (LSP)\* | 言語固有のサーバーを通じたリアルタイムのコードインテリジェンス | 一度設定すれば常時利用可能 | 型付き言語におけるシンボルレベルのナビゲーションと自動エラー検知 | 自動で動くものだと思い込む |
| MCP サーバー | 外部ツールやデータへの接続 | 一度設定すれば常時利用可能 | それ以外の方法では届かない社内ツールへのアクセスを Claude に与える | 基本が機能する前に MCP 接続を作り込む |
| サブエージェント\* | 特定タスク向けの別 Claude インスタンス | 呼び出されたとき | 探索と編集の分離、並列作業 | 同じセッションで探索と編集を一緒に走らせる |
| \*LSP はプラグインレイヤー経由でアクセスします。サブエージェントは設定された拡張ポイントというよりも、移譲のための能力です。 | | | | |

## 成功したデプロイメントから見えた 3 つの設定パターン

大規模コードベース向けに Claude Code をどう設定するかは、そのコードベースがどう構造化されているかに大きく依存します。それでも、私たちが観察したデプロイメントには 3 つのパターンが一貫して現れました。

### コードベースを大規模でも辿れるようにする

Claude が大規模コードベースで力を発揮できるかどうかは、適切なコンテキストを見つけられるかどうかに縛られます。毎セッションに詰め込みすぎたコンテキストは性能を悪化させ、少なすぎれば Claude は手探りで歩くしかありません。最も効果的なデプロイメントは、コードベースを Claude にとって読みやすくするための先行投資を行っています。いくつかのパターンが一貫して現れます。

- **CLAUDE.md ファイルを軽くレイヤー化して保つ。** Claude はコードベースを移動しながら、これらを加算的に読み込みます——全体像のためのルートファイル、ローカルな慣習のためのサブディレクトリファイルです。ルートファイルはポインターと致命的な落とし穴だけにとどめ、それ以外はノイズに変わっていきます。
- **リポジトリのルートではなく、サブディレクトリで初期化する。** Claude は、タスクに本当に関係のあるコードベースの一部にスコープを絞られたときに最もうまく動きます。モノレポではツールがしばしばルートアクセスを前提とするため、これは直感に反するかもしれません。しかし、Claude はディレクトリツリーを自動的に遡って通り道にある CLAUDE.md をすべて読み込むので、ルートレベルのコンテキストが失われることはありません。
- **テストや lint のコマンドをサブディレクトリごとにスコープする。** Claude が 1 つのサービスを変えただけでフルスイートを走らせると、タイムアウトを引き起こし、関係のない出力でコンテキストを浪費します。サブディレクトリレベルの CLAUDE.md には、コードベースのその部分に適用されるコマンドを書いておくべきです。これは、ディレクトリごとに独自のテストとビルドのコマンドを持つサービス志向のコードベースではうまく機能します。コンパイル系言語のモノレポでディレクトリをまたぐ深い依存がある場合、サブディレクトリごとのスコープ化は難しく、プロジェクト固有のビルド設定が必要になるかもしれません。
- **`.`**`ignore` **ファイルを使って、生成ファイル、ビルド成果物、サードパーティコードを除外する。** `.claude/settings.json` の `permissions.deny` ルールをコミットしておけば、除外設定がバージョン管理下に置かれ、チーム全員が自分で設定することなく同じノイズ削減を享受できます。一部のコードベースでは、生成ファイル自体が開発作業の対象になっています。コードジェネレーターを担当する開発者は、プロジェクトレベルの除外をローカル設定で上書きすることで、チームの他のメンバーに影響を与えずに済みます。
- **ディレクトリ構造に頼れないときはコードベースマップを構築する。** コードが従来型のディレクトリ構造に集約されていない組織では、リポジトリのルートに、トップレベルの各フォルダとそこに何があるかの一行説明を並べた軽量な Markdown ファイルを置いておくと、Claude にファイルを開く前にスキャンできる目次を与えられます。トップレベルのフォルダが何百もあるようなコードベースでは、レイヤー化したアプローチが最もうまく機能します——ルートのファイルには最上位の構造だけを書き、その次のレベルの詳細はサブディレクトリの CLAUDE.md に書いて、Claude がツリーを移動する中でオンデマンドに読み込まれるようにします。よりシンプルなケースでは、Claude に参照させたい具体的なファイルやディレクトリを @ メンションするだけで同じ役割を果たせます。
- **LSP サーバーを動かして、Claude が文字列ではなくシンボルで検索できるようにする。** 大規模コードベースでよくある関数名を grep すると数千件のマッチが返ってきて、Claude はどれが本命かを判断するためにファイルを開いてコンテキストを燃やすことになります。LSP は同じシンボルを指す参照だけを返すので、Claude が何かを読む前にフィルタリングが終わります。これを動かすには、対象言語向けの[コードインテリジェンスプラグイン](https://code.claude.com/docs/en/discover-plugins#code-intelligence)と対応する言語サーバーのバイナリをインストールする必要があります。利用可能なプラグインやトラブルシューティングは Claude Code のドキュメントが扱っています。

**ひとつだけ注意**: 階層的な CLAUDE.md アプローチですら破綻するエッジケースもあります。たとえば数十万のフォルダと数百万のファイルを持つコードベースや、Git ではないバージョン管理上で動くレガシーシステムです。これらの課題については、このシリーズの今後の回で取り上げます。

### モデルの知能の進化に合わせて CLAUDE.md ファイルを能動的にメンテナンスする

モデルが進化するにつれて、現行モデル向けに書いた指示が将来のモデルに対しては逆効果になることがあります。以前は Claude がつまずいていたパターンを導くために書かれた CLAUDE.md は、次のモデルが出てくると不要になるか、むしろ足を引っ張る制約になることがあります。たとえば、すべてのリファクタを単一ファイルの変更に分けるよう Claude に指示する CLAUDE.md のルールは、古いモデルを軌道に乗せるのに役立っていたかもしれませんが、それをうまくこなせる新しいモデルが調整されたクロスファイルの編集を行うのを妨げてしまうでしょう。

特定のモデルの制約——モデルの推論の限界であれ、Claude Code 自体のツール側の限界であれ——を補うために作られた skill や hook は、その制約がなくなれば余計なものになります。Perforce のコードベースで p4 edit を強制するためにファイル書き込みをインターセプトしていた hook は、Claude Code がネイティブの Perforce モードを追加した時点で冗長になりました。

チームは 3〜6 か月ごとに意味のある設定レビューを行うつもりでいるべきですし、メジャーなモデルリリースの後で性能が頭打ちに感じられたときにもレビューを行う価値があります。

### Claude Code の管理と普及のオーナーシップを割り当てる

技術的な設定だけでは普及は進みません。これをうまくやった組織は、組織レイヤーにも投資していました。

最も速く広まったロールアウトには、広範なアクセス開放に先立って専門のインフラ投資がありました。小さなチーム、ときには一人だけが、開発者が最初に Claude Code に触れたときにはすでにそれが自分のワークフローに馴染むよう、ツールを配線しておきました。ある企業では数人のエンジニアがプラグインと MCP の一式を作り、初日から使える状態にしました。別の企業では、AI コーディングツールの管理に専念するチーム全体が、ロールアウト開始前にインフラを整えていました。どちらの場合も、開発者の最初の体験は不満ではなく生産的なものになり、そこから普及が広がっていきました。

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a04e25f1984beb50dc5525b_fig2-phases-of-claude-code-rollout-v1%402x.png)

この仕事を今日担っているチームは、開発者体験や開発者生産性の傘下にあるケースが多く、新しいエンジニアのオンボーディングや開発者向けツールの構築を担当するファンクションがそれにあたります。いくつかの組織で新しく現れている役割が「エージェントマネージャー」です——Claude Code のエコシステム管理に専念する、PM とエンジニアのハイブリッド職です。専任チームのない組織にとって、最小限の形は DRI を立てることです——Claude Code の設定、設定・パーミッションポリシー・プラグインマーケットプレイス・CLAUDE.md の慣習に関する判断権限を持ち、それらを最新に保つ責任を負う、一人の担当者です。

ボトムアップの普及は熱意を生みますが、機能するものを中央集権化する誰かがいなければ、簡単にバラバラになります。誰か個人かチームを置いて、適切な Claude Code の慣習（標準化された CLAUDE.md の階層構造や、厳選された skill とプラグインのセットなど）を組み立て、布教させる必要があります。そうした仕事がなければ、知見は部族的なまま留まり、普及は頭打ちになります。

大きな組織、特に規制業種においては、ガバナンスに関する問いが早い段階で出てきます——どの skill とプラグインを利用可能にするかを誰が管理するのか、数千人のエンジニアが同じものを独立に作り直すのをどう防ぐのか、AI 生成のコードが人間が書いたコードと同じレビュープロセスを通ることをどう担保するのか、などです。これに早めに対処するために、私たちは承認済みの skill セット、必須のコードレビュープロセス、限定された初期アクセスから始め、信頼が積み上がるにつれて広げていくことを推奨しています。

私たちは、エンジニアリング、情報セキュリティ、ガバナンスの代表者を早い段階から集めてクロスファンクショナルなワーキンググループを作り、要件を共に定義し、ロールアウトのロードマップを構築している組織で、最もスムーズなデプロイメントを観察してきました。

## これらのパターンをあなたの組織に適用する

Claude Code は、エンジニアがコードベースの主要な貢献者であり、リポジトリが Git を使い、コードが標準的なディレクトリ構造に従っている、という従来型のソフトウェアエンジニアリング環境を念頭に設計されています。ほとんどの大規模コードベースはこの型に収まりますが、大きなバイナリアセットを抱えるゲームエンジン、変則的なバージョン管理を使う環境、エンジニア以外がコードベースに貢献するような非伝統的な構成では、追加の設定作業が必要になります。私たちのガイダンスは従来型のセットアップを前提としており、ここに記したパターンは多くのお客様で機能してきました。残された複雑さには、あなたのコードベース、ツール、組織に固有の判断が必要です。そこは Anthropic の Applied AI チームがエンジニアリングチームと直接協働し、これらのパターンをあなたの組織固有の要件に翻訳していく領域です。
