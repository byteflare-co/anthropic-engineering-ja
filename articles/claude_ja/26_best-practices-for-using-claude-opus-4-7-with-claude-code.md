# Claude Code で Claude Opus 4.7 を使いこなすためのベストプラクティス

[Opus 4.7](https://www.anthropic.com/news/claude-opus-4-7) は、コーディング、エンタープライズワークフロー、長時間のエージェンティックタスクにおいて、現時点で最も高性能な一般提供モデルです。Opus 4.6 と比べて曖昧さの処理に優れ、バグの発見やコードレビューの能力が大幅に向上し、セッションをまたいだコンテキストの保持がより確実になり、指示が少なくても曖昧なタスクを推論で処理できるようになりました。

[ローンチのお知らせ](https://www.anthropic.com/news/claude-opus-4-7)で触れたとおり、2 つの変更点——トークナイザーの更新と、高い effort レベルでの（特に長いセッションの後半ターンにおける）思考量の増加——がトークン使用量に影響します。そのため、Opus 4.6 を Opus 4.7 に置き換える際は、最高のパフォーマンスを引き出すために若干のチューニングが必要になることがあります。プロンプトやハーネスにいくつかの工夫を加えるだけで、大きな違いが生まれます。

この記事では、何が変わったのか、そして Claude Code で Opus 4.7 を最も効果的に活用する方法を解説します。

## インタラクティブなコーディングセッションの構成

Opus 4.7 のトークン使用量と振る舞いは、ユーザーターンが 1 回のより自律的な非同期コーディングエージェントとして使う場合と、ユーザーターンが複数回あるよりインタラクティブな同期コーディングエージェントとして使う場合で異なります。インタラクティブな設定では、ユーザーターンの後でより多くの推論を行います。これにより長いセッション全体での一貫性、指示への追従性、コーディング品質が向上しますが、トークンの使用量が増える傾向があります。

Claude Code で Opus 4.7 を最大限に活用するには、1 行ずつ指示するペアプログラマーとしてではなく、仕事を委任する優秀なエンジニアとして Claude を扱うのが効果的です。

- **タスクは最初のターンで明確に指定する。** 意図、制約条件、受け入れ基準、関連するファイルの場所を含むよく練られたタスク記述があれば、Opus 4.7 はより高品質な出力を返します。何度ものターンにわたって段階的に伝える曖昧なプロンプトは、トークン効率と、場合によっては全体的な品質の両方を低下させがちです。
- **ユーザーインタラクションの回数を減らす。** ユーザーターンが増えるたびに推論のオーバーヘッドが発生します。質問はまとめて行い、モデルが作業を止めずに進められるだけのコンテキストを与えてください。
- **適切な場面で [auto mode](https://claude.com/blog/auto-mode) を使う。** 頻繁に確認しなくてもモデルに安全に実行を任せられるタスクでは、auto mode を使うとサイクルタイムが短縮されます。コンテキストを最初にすべて提供した上での長時間タスクに特に適しています。auto mode は現在、Claude Code Max ユーザー向けにリサーチプレビューとして提供されており、Shift+Tab で切り替えられます。
- **タスク完了時の通知を設定する。** タスクが終わったらサウンドを鳴らすよう Claude に依頼すれば、Claude 自身がフックベースの通知を作成できます。

## Opus 4.7 の推奨 effort 設定

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69e1088ee4f61fecf4983993_Opus-4-7-Agentic-coding-performance-by-effort-Break.png)

Claude Code における Opus 4.7 のデフォルト effort レベルは `xhigh` に変更されました。これは `high` と `max` の間に位置する新しい effort レベルで、難しい問題における推論とレイテンシーのトレードオフをより細かく制御できます。ほとんどのエージェンティックなコーディング作業、特に API やスキーマの設計、レガシーコードの移行、大規模コードベースのレビューといった知的負荷の高いタスクには xhigh を推奨します。

各 effort レベルに関する追加のガイダンスを示します。

- **`medium` と `low`**: コスト重視、レイテンシー重視、またはスコープが限定された作業に適しています。高い effort レベルと比べて難しいタスクでの性能は劣りますが、同じ effort レベルの Opus 4.6 よりは依然として優れた性能を発揮します——場合によってはより少ないトークンで実現します。
- **`high`**: 知的性能とコストのバランスが取れた設定です。複数セッションを並行して実行する場合や、品質を大きく落とさずにコストを抑えたい場合に選んでください。
- **`xhigh`（デフォルト、推奨）**: ほとんどのコーディングおよびエージェンティック用途に最適な設定です。長時間のエージェンティック実行で max が引き起こしがちな過剰なトークン使用を避けつつ、高い自律性と知的性能を備えています。
- **`max`**: 真に難しい問題で追加の性能を引き出しますが、収穫逓減が見られ、過度に考えすぎる傾向があります。評価（eval）でモデルの最大能力を測定する場合や、きわめて知的負荷が高くコストを気にしないユースケースに限定してお使いください。

新しいモデルにアップグレードする場合は、以前の設定をそのまま引き継ぐのではなく、effort レベルを試行してみることをお勧めします。同じタスク内で effort レベルを切り替えることで、トークン使用量と推論をより効果的に管理できます。

Opus 4.7 のデフォルト effort レベルを xhigh に設定したのは、ほとんどのコーディングタスクに最適だと考えているためです。既存の Claude Code ユーザーで effort レベルを手動で設定していない場合は、自動的に `xhigh` にアップグレードされます。手動での調整はこれまでどおり可能です。

## adaptive thinking の活用

**Opus 4.7 では、固定の thinking バジェットを指定した Extended Thinking はサポートされていません。** 代わりに、Opus 4.7 は adaptive thinking を提供します。これにより各ステップで thinking がオプションとなり、コンテキストに応じてモデル自身がより多くの思考を行うべきかどうかを判断できるようになります。シンプルなクエリには素早く応答し、思考が有益でないステップでは thinking をスキップし、最も効果的な箇所に thinking トークンを投入できます。エージェンティックな実行全体を通じて見ると、これにより応答が高速化し、ユーザー体験が向上します。

今回のリリースでは adaptive thinking が大幅に改善されており、特に Opus 4.7 は過度な思考に陥りにくくなっています。

thinking の頻度をより細かく制御したい場合は、プロンプトで直接指示してください。

- **thinking を増やしたい場合**: 「回答する前に慎重にステップバイステップで考えてください。この問題は見た目より難しいです。」のような指示を試してください。
- **thinking を減らしたい場合**: 「深く考えるよりも素早く回答することを優先してください。迷ったら直接回答してください。」のような指示を試してください。トークンは節約できますが、難しいステップでは精度が低下する可能性があります。

## 把握しておくべき振る舞いの変更点

Opus 4.6 から 4.7 にかけて、いくつかのデフォルトの振る舞いが変更されています。旧モデルに合わせてプロンプトやハーネスを入念にチューニングしていた場合は、把握しておくとよいでしょう。

**応答の長さがタスクの複雑さに応じて調整されるようになりました。** Opus 4.7 は Opus 4.6 ほどデフォルトで冗長ではありません。シンプルな問い合わせにはより短い回答が、自由度の高い分析にはより長い回答が返ってきます。特定の長さやスタイルに依存するユースケースの場合は、プロンプトで明示的に指定してください。望むスタイルの具体例を示すほうが、「〜しないでください」のような否定的な指示よりも効果的です。

**モデルがツールを呼び出す頻度が下がり、代わりにより多く推論するようになりました。** これは多くの場合により良い結果をもたらします。ツールをもっと使ってほしい場合（たとえば、エージェンティック作業中により積極的に検索やファイル読み取りを行ってほしい場合）は、ツールをいつ、なぜ使うべきかを明示的に記述するガイダンスを提供してください。

**デフォルトでサブエージェントの生成が控えめになりました。** Opus 4.7 はサブエージェントに作業を委任するタイミングをより慎重に判断する傾向があります。並列サブエージェントが有効なユースケース（たとえば、複数のファイルや独立した項目を並行して処理する場合）では、その旨を明示することを推奨します。例:

Do not spawn a subagent for work you can complete directly in a single response (e.g., refactoring a function you can already see). Spawn multiple subagents in the same turn when fanning out across items or reading multiple files.

## 次のステップ

Opus 4.7 は、以前のモデルよりも長時間実行タスクで優れた性能を発揮します。そのため、複雑なマルチファイル変更、曖昧なデバッグ、サービス全体のコードレビュー、多段階のエージェンティック作業など、これまで人間の監督がボトルネックだったタスクに適しています。

effort を `xhigh` に設定したまま、最初のターンでどこまで到達できるか試してみてください。
